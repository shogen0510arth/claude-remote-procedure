Claude Code で
PowerPoint / Excel を作る運用ガイド

ARTHの資料作成ワークフロー（vibe coding / ちょこちょこ開発）。常時稼働のMac miniを計算ノード、MacBook Airを携帯クライアントとして、テンプレート準拠の資料生成を「翌週も続きから」回すための運用ガイド。

作成日：2026-07-19 PowerPoint Excel

目次

    0

    前提

    • プラン：Claude Code は Pro / Max / Team / Enterprise が必要（無料プラン不可）。
    • 実行場所：常時稼働の Mac mini を計算ノード、MacBook Air を携帯クライアントとする想定。
    • 中身のエンジン：PowerPoint / Excel の生成は Claude Code が python-pptx / openpyxl 等のコードを書いて実行する。チャット版と同じ仕組みで、差は「どこで動くか」だけ。
    • チャット版との使い分け：単発・たたき台はチャット版、テンプレ準拠の定型資料や繰り返し更新するものは Claude Code。Claude Code はフォルダもスクリプトも手元に残るので「翌週も続きから」ができる。
    1

    インストール

    BASH — macOS / Linux / WSL
    curl -fsSL https://claude.ai/install.sh | bash

    確認：

    BASH
    claude --version
claude doctor      # 設定・認証の診断

    ターミナルが不要な Desktop アプリ版もある（同じ Claude Code を GUI で使える）。

    ドキュメントスキルの導入（最重要）

    生成物の質を最も左右する

    素の python-pptx で作らせるより、Anthropic公式のドキュメントスキルを入れると、テンプレート追従・レイアウト・検証（サムネイル生成）まで自動でやってくれる。

    Claude Code 内で：

    CLAUDE CODE
    /plugin marketplace add anthropics/skills
/plugin install document-skills@anthropic-agent-skills
    • pptx / docx / xlsx / pdf のスキルが有効化され、「pptxスキルでこの内容のスライドを作って」等で呼べる。
    • スキルは「テンプレのテキスト抽出 → 差し替え → サムネイル生成 → 検証」の流れで作るため、素のコード生成より仕上がりが安定する。
    • プロジェクト固有にしたいなら .claude/skills/<name>/SKILL.md を置く方法もある（例：ARTH独自のブランドPPTX適用手順をスキル化すれば、毎回プロンプトで指定する必要がなくなる）。
    2

    プロジェクトフォルダの作り方

    方針：1つの ARTH 親フォルダ + クライアント別サブフォルダ。
    Claude Code は起動フォルダから親方向へ CLAUDE.md を遡ってすべて読み込むため、共通ルール（書式・テンプレ）を1箇所で管理しつつ、案件ごとの文脈を足せる。

    ~/arth/ ├── CLAUDE.md # 全社共通ルール（必ず読まれる） ├── templates/ │ ├── arth-format.pptx # PowerPoint の会社フォーマット │ └── arth-format.xlsx # Excel のひな型 ├── assets/ # ロゴ・画像素材 └── clients/ ├── yoshida/ │ ├── CLAUDE.md # ヨシダ案件固有の文脈 │ ├── source/ # 参照する過去資料・データ・LINEログ │ └── output/ # 生成物 └── <other-client>/ └── ...
    • テンプレは「マスタースライド・フォント・配色・ロゴが仕込まれた既存の .pptx / .xlsx」を置く。ゼロから作らせず、これを複製させるのがコツ。
    • 1クライアントが複数案件を持つ場合は yoshida/ の下にさらに案件サブフォルダを切る（同じ仕組みで階層化）。
    完全分離すべきケース

    競合する取引先を同時担当 / NDAが厳しい / 担当者が別 / Git履歴・アクセス権を分けたい → そのクライアントだけ独立フォルダ（または別リポジトリ）にする。

    テンプレートの作り込み（仕上がりを決める前提条件）

    スキルを入れても、テンプレの作り方が悪いと追従できない。ここが甘いと何度作り直しても揃わない。

    • スライドマスター/レイアウトを正しく作る：手描きの1枚もの .pptx ではなく、PowerPoint のスライドマスター表示で「表紙」「本文」「図表」等のレイアウトとプレースホルダを定義しておく。Claude はこのレイアウトを複製して中身を流し込む。
    • 代表スライドを2〜3枚同梱：完成イメージの見本を templates/ に入れておくと、トーン・構成を掴みやすい。
    • フォントを Mac mini にインストール：ブランドフォントが未インストールだと生成物が別フォントにフォールバックする。生成ノード（Mac mini）側に必ず入れる。
    • Excel も同様：数式・条件付き書式・シート構成をひな型で固めておく。

    CLAUDE.md の書き方（会社フォーマット準拠の肝）

    CLAUDE.md は毎セッション自動で読まれる常設の指示書。ここに書式ルールを書けば毎回説明不要。

    MARKDOWN — CLAUDE.md
    # ARTH 資料作成ルール

## 基本方針
- 新規スライド/シートは templates/ のマスター・配色・フォントを継承する
- 過去の書式は各クライアントの source/ を参照して踏襲する
- ロゴ等は assets/ から使う
- 完成物は各クライアントの output/ に日付付き（YYYYMMDD_名称）で保存する

## 参照ルール
- 案件の背景・条件は、まず該当クライアントの source/ と Gmail の該当ラベルを確認する
- 予定・提出期日は Google カレンダーを確認する
- このクライアントの経緯は source/line-log.md も参照する

## トーン
- 投資家・外部企業向け。数値は根拠を明示。1スライド1メッセージ。

## 生成方法
- PowerPoint は python-pptx でテンプレを読み込み、レイアウトを複製して作る
- Excel は openpyxl。数式・書式はひな型を踏襲
- 白紙から作らない
    CLAUDE.md の階層ルール

    親（~/arth/CLAUDE.md）と子（clients/yoshida/CLAUDE.md）は両方ロードされ、競合時はより下層（案件固有）が優先される。~/.claude/CLAUDE.md は全プロジェクト共通のグローバル設定。

    セッションの立て方

    • 必ずクライアントのサブフォルダから起動する（親の ~/arth/ 直下からは起動しない）。
    BASH
    cd ~/arth/clients/yoshida
claude

    → 親の共通ルール（書式）を継承しつつ、セッションは1クライアントに閉じる。親から起動すると他クライアントのファイルまで横断でき、情報が混ざるリスクがある。

    • セッションは Claude Code が動いている間だけ状態を保持する。フォルダ・スクリプトはローカルに残るので翌週も続きから可能。
    • Mac mini で回すコツ：SSH + tmux で Mac mini に入って起動すると、MacBook Air を閉じても走り続ける（既存の Tailscale + SSH + tmux 環境をそのまま活用）。

    Gmail / Google カレンダーとの連携

    Claude Code は独自にMCPを登録する必要がある（チャット版のコネクタ設定とは別物）。Google は公式のリモートMCPサーバーを提供している。

    登録コマンド

    BASH
    claude mcp add --transport http gmail   https://gmailmcp.googleapis.com/mcp/v1
claude mcp add --transport http gcal    https://calendarmcp.googleapis.com/mcp/v1
claude mcp add --transport http gdrive  https://drivemcp.googleapis.com/mcp/v1

    追加後、Claude Code 内で /mcp を実行 → ブラウザで OAuth 認証。

    認証の注意

    • Google公式のWorkspace MCPは OAuth 2.0 で、Google Cloud プロジェクトと OAuth クライアントID/シークレットの用意が必要（現状 Developer Preview）。ここが唯一やや手間。
    • OAuth設定が煩雑なら、コミュニティ製の統合 Google Workspace MCP（npx / uvx で立てるタイプ）も選択肢。
    トークン消費に注意

    MCPを載せると多数のツールが常時コンテキストに載りトークンを消費する。使うサービスだけ入れる。

    使い方（Gmail 特定ラベル）

    • Gmail検索クエリで絞れる：label:クライアント-ヨシダ など。該当ラベルのスレッドだけをオンデマンドで読ませる。
    書き込み系は後回し

    書き込み系（送信・下書き）は必要になるまで有効化しない方が安全。メール本文はプロンプトインジェクションの経路になりうる。

    使い方（カレンダー）

    予定の一覧・作成・更新が可能。資料の提出期日や打ち合わせ日程を Claude に確認・反映させられる。

    「常時参照」の考え方（重要）

    Claude Code は裏で常駐監視しない。実現できるのは次の2層。

    • オンデマンド層（MCP / ローカル）：呼べばいつでも読める。Gmailラベル・カレンダー・Drive・ローカルはここ。
    • 取り込み層（n8n / cron / launchd）：常に最新にしたいもの（新着メール等）はスケジュール実行でローカルかNotionにコピーを溜め、Claudeはそのコピーを読む。

    CLAUDE.md に「まず該当ラベルとカレンダーを見る」と書けば既定で参照しにいく。モデル内で「常時参照」に一番近い形。

    5

    Mac mini のフォルダの連携

    目的：Mac mini（Claude Codeノード）が参照資料を確実にローカルで読める状態にする。MacBook Air は最適化ON（軽量）のまま。

    AiCloud +「ダウンロードを維持」（最小変更・推奨）

    1. 資料フォルダを iCloud Drive 内に置く
    2. Mac mini の Finder でそのフォルダを右クリック → 「ダウンロードを維持（Keep Downloaded）」
    • そのフォルダは退避されず常にローカルに実体を持つ。
    • Airで作って同期されてきた新規ファイルも自動で維持対象（フォルダ単位のピンは既存＋新規の両方に効く）。
    • ピンはデバイスごと。miniで設定してもAirには影響しない（Airは軽量のまま）。
    • 確認：Finderのアイコンで判定（下向き矢印＝維持済み、雲＝まだローカルに無い）。
    • 注意：Air作成 → アップ → mini取得の同期ラグあり。作成直後に叩くワークフローは少し待つ設計にする。

    BMac mini だけ Optimize を OFF（最も堅牢）

    • システム設定 → Apple アカウント → iCloud → iCloud Drive →「Macストレージを最適化」を OFF
    • iCloud Drive全部が常にローカル。ピン不要でメンテナンスフリー。
    • 設定はそのMac限定。Airは無関係（ONのまま）。
    • ディスクに余裕がある専用ノード向き。

    CGoogle Drive にミラー

    • Google Drive for desktop → 環境設定 →「パソコンのフォルダ」で対象フォルダを追加 → 双方向ミラー。
    • Claude Codeはミラーされたローカルフォルダを直接読める（MCP不要）。Drive MCP経由でも可。
    二重同期しない

    同じフォルダを iCloud と Google Drive の両方で同期しない（競合する）。どちらか一方に寄せる。

    補足：編集させたいファイルの置き場所

    • Claude Codeで .pptx / .xlsx を編集させたいなら、ローカル（最良）か Google Drive（Drive MCPがbase64でDL→編集→再アップ可）に置く。
    Notion は編集不可

    公式Notion MCPはファイルアクセスを公開しておらず、Notion上の .pptx / .xlsx をDL→編集→書き戻す一連はできない。Notionはテキスト知識の置き場に限定する。
    （参考：Notion 3.6 以降、Notion自身のエージェントは pptx/xlsx の読み書きに対応。ただしこれは Claude Code とは別ツール）

    git リポジトリの注意

    iCloud Drive はアクティブな git リポジトリには不向き。コードは ~/Developer/ 等に置く（既存運用通り）。この連携は「参照資料置き場」用。

    LINE の過去ログのコンテクスト化 + bot の入れ方

    制約（重要）

    LINE の Messaging API は、公式アカウント（bot）を招待したグループで、bot参加後のメッセージをWebhookで受け取る方式。

    • 過去にさかのぼって履歴を取得するAPIは無い。グループ名・メンバーIDは取れるが、スクロールバックは取れない。
    • botをグループに入れるには設定で「Allow bot to join group chats」を有効化（既定OFF）。1グループに入れる公式アカウントは同時に1つ。

    結論（2つに分ける）

    • 参加以降の新規発言：botで拾ってファイル/DBに溜め、Claude Codeがそれを読む（取り込み層）。
    • 参加前の過去ログ：スマホのLINEでトーク履歴を .txt エクスポート し、クライアントの source/ に置く → Claude Codeが読む（手動）。

    bot導入手順（概要）

    1. LINE Developers コンソールで Provider を作成 → Messaging API チャネルを作成
    2. チャネルの Messaging API タブで「Allow bot to join group chats」を有効化
    3. Channel access token を発行
    4. Webhook URL を設定（受信サーバー）。既に n8n があるので n8n の Webhook ノードで受けるのが早い
    5. 対象グループに公式アカウントを招待
    6. n8n で受信メッセージ（group_id, user_id, text, timestamp）を整形 → ファイル / Notion / DB に追記
    7. Claude Code はその蓄積ファイル（例：source/line-log.md）を参照

    ※ この「LINEグループの文脈を取り込む」構成は、既存のメール自動化（n8n）と同じパターンで拡張できる。

    過去ログのコンテクスト化のコツ

    • エクスポートした .txt はそのままだと冗長。日付・発言者・要点に整形して置くと、参照時の精度とトークン効率が上がる。
    • CLAUDE.md に「このクライアントの経緯は source/line-log.md を参照」と明記しておく。

    コンテクストを読み込みながら指示を出すプロンプト例

    前提：cd ~/arth/clients/yoshida && claude で起動済み（共通ルール＋ヨシダ文脈がロード済み）。

    例1：過去資料の書式を踏襲して新規作成

    プロンプト
    source/ の過去の投資家向け資料を参考に、西伊豆プロジェクトの提案資料を
templates/arth-format.pptx のフォーマットで作って。
Gmail の label:クライアント-ヨシダ の直近スレッドから条件（金額・スケジュール）を拾って反映。
提出期日は Google カレンダーで確認して表紙に入れて。
完成したら output/ に日付付きで保存。

    例2：Excelモデルの更新

    プロンプト
    source/ の前四半期の実績データと templates/arth-format.xlsx をもとに、
今四半期の収支シートを作成。数式と書式はひな型を踏襲。
source/line-log.md で決まった前提条件があれば反映して。

    例3：メール文脈だけ先に要約させる（作成前の確認）

    プロンプト
    Gmail の label:クライアント-ヨシダ の未読・直近1週間を読んで、
資料に反映すべき論点だけ箇条書きで出して。まだ資料は作らないで。

    → 先に文脈を確定させてから作成に進むと精度が上がる（収集 → 確認 → 生成の流れ）。

    例4：既存資料の修正

    プロンプト
    output/20260719_西伊豆提案.pptx の3枚目の図を、
source/最新実績.xlsx の数値で差し替えて。配色はテンプレのまま。

    プロンプトのコツ

    • 参照元を明示：「source/ を見て」「label:xxx から」「カレンダーで」と、どこを読むか具体的に指定する。
    • XMLタグ / 明確な境界：複雑な指示はタグや箇条書きで区切ると安定（XMLが最も確実な境界マーカー）。
    • 一度に全部やらせない：文脈収集 → 確認 → 生成 → 修正、と段階を分ける。
    • 確認を挟む：「まだ作らないで、論点だけ」で暴走を防ぐ。書き込み系（メール送信等）には特に承認ゲートを設ける。

    過去の会話の扱い + 品質・信頼性のための追加設定

    8-1. 「過去の会話」を文脈にする

    Claude Code のセッションは、新しいセッションに自動では引き継がれない。過去の会話を根拠にするには次のいずれか。

    • 決定事項をファイルに外部化（推奨）：各クライアントに source/decisions.md を置き、会話の最後に「この会話の決定事項を source/decisions.md に追記して」と指示する。次回以降は CLAUDE.md の参照ルールでここを読ませる。最も確実で、人間も後から追える。
    • セッションを再開：claude --continue（直近を継続）/ claude --resume（一覧から選択）で前回の文脈を引き継ぐ。
    • 使い分け：長期の蓄積は decisions.md、短期の続きは --resume。

    8-2. 確認（プレビュー）ループ

    • Claude Code は .pptx を画面にレンダリングできない。pptxスキルはサムネイル画像を生成するので、それで素早く目視できる。
    • 必要なら「PDFに書き出して」と頼む（LibreOffice の headless 変換等）。PowerPoint を開かずに確認でき、反復が速い。

    8-3. 情報の優先順位（source of truth）

    • メール・LINE・過去資料で数値が食い違うことがある。CLAUDE.md に優先順位を明記する。
      例：「金額・日程は 最新のメール > 過去資料。矛盾があれば指摘してから進める」
    • これは「単一マスターを先に決める」原則と同じ発想。矛盾を黙って埋めさせない。

    8-4. バージョン管理とバックアップ

    • ~/arth/（テンプレ・CLAUDE.md・スクリプト・成果物）は git 管理して GitHub に push。テンプレやルールは資産なので必ず版管理する。
    • iCloud は git に不向きなので、~/arth/ 自体は iCloud 外（~/Developer/ 等）に置くのが無難。大きな参照資料だけを iCloud / Drive 経由で各 source/ に配置・同期する、と役割を分ける（§5参照）。

    8-5. 摩擦を減らす設定

    • .claude/settings.json で定型ツールを許可しておくと、毎回の確認プロンプトが減り vibe coding が快適になる。ただし書き込み系（メール送信等）は承認を残す。
    • コンテキスト肥大を避ける：「全部読んで」ではなく対象ファイルを指定。無関係な作業に切り替える時は /clear でリセット、関連作業の圧縮は /compact。
    • .claudeignore で巨大・無関係ディレクトリを除外。

    8-6. 新規クライアント立ち上げチェックリスト

    1. clients/<name>/ を作成し、source/ output/ を用意
    2. CLAUDE.md を親からコピーして案件固有情報を追記
    3. 過去資料を source/ に配置（または iCloud / Drive から）
    4. 参照するメールラベルを決める（例：label:クライアント-<name>）
    5. LINEグループがあれば bot 参加 + ログ蓄積先を決める
    6. cd clients/<name> && claude で起動し、テスト生成 → 微修正でテンプレ/CLAUDE.md を調整

    まとめ（役割分担の全体像）
