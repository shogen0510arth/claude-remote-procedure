Claude Code
プラットフォーム比較と機能ガイド

根本のClaude Codeエンジンは同じだが、操作画面と一部機能が異なる。大まかには、CLIが最も高機能、IDEがコード確認・修正に強く、Macアプリが最も初心者向け、iPhoneアプリは遠隔操作用。

CLI版 IDE版 Macアプリ版 iPhoneアプリ

目次

    ◎

    概要

    Claude Codeには複数の操作画面（プラットフォーム）がある。動いている根本のエンジンは同じだが、それぞれ得意分野が異なる。まず全体像を1文でまとめると次のとおり。

    • CLI版：最も高機能。設定・自動化・サーバー運用に強い。
    • IDE版（VS Code等）：コード確認・修正に強い。
    • Macアプリ版：最も初心者向け。GUIで総合的に開発できる。
    • iPhoneアプリ：遠隔操作用。指示・確認・承認が中心。
    公式リファレンス

    各プラットフォームの違いは 公式のプラットフォーム比較 にもまとまっている。

    CLI版

    最も機能が揃っている。

    できること

    • claude remote-control --capacity 3などのサーバー起動
    • claude -c、claude --resumeによるセッション復旧
    • MCPサーバーの追加・設定
    • すべてのスラッシュコマンドと設定
    • シェルスクリプトや定期実行
    • Agent SDKを使った独自システムへの組み込み
    • Git worktreeを使った高度な並列開発
    • 詳細なログ確認とトラブルシューティング

    一方、コードや差分を目で確認するにはあまり向かない。

    今回の運用でのCLIの役割

    CLIは日常的なプロンプト入力ではなく、次のためだけに使えば十分。

    • Mac miniでRemote Controlを起動する
    • プロジェクト別サーバーを起動する
    • 終了したセッションを復旧する
    • 初期設定やトラブル対応を行う

    CLIはスクリプト・Agent SDKを含む最も完全な操作面である。Claude Code CLI公式リファレンス

    3

    IDE版

    主にVS Code、Cursor、JetBrains系IDEで使う形。強みは、コードを自分でも見たり編集したりしながらClaudeを使えること。

    • 選択した行をそのままClaudeへ渡せる
    • 変更前後の差分を横並びで確認できる
    • Claudeの修正案を一部変更してから承認できる
    • 計画書へ行単位でコメントできる
    • エラーや警告をClaudeと共有できる
    • ターミナルの出力をコピーせず参照できる
    • 複数セッションをタブで開ける
    利用できるコマンドはCLIの一部

    IDE版で利用できるコマンドはCLIの一部である。MCPの新規追加などは、VS Code内のターミナルでCLIを併用する必要がある。CLIとVS Code拡張は設定やローカルの会話履歴を共有できる。

    向いている場面

    • Claudeが変更したコードを細かく確認したい
    • 自分でも少しコードを直したい
    • 特定のファイル・行について質問したい
    • エラー箇所を目で見ながら修正したい

    プログラミング初心者でも、開発が進んでから「どこを変えたのか確認したい」と思うようになった段階で導入すると便利。VS Code版公式ドキュメント

    Macアプリ版

    Claude Desktopの「Code」タブ。現在はかなり高機能で、単なるチャット画面ではない。

    • 複数セッションをサイドバーで管理
    • チャット、差分、ブラウザ、ターミナル、ファイルを並べて表示
    • アプリのプレビュー
    • ファイルの簡単な直接編集
    • 画像・PDF等のドラッグ＆ドロップ
    • Gitの差分確認
    • PRの進捗確認
    • MCP・コネクタ・プラグイン管理
    • 複数セッションの自動Git分離
    • スマホから作業を送るDispatch
    • Remote Control
    • ローカル、クラウド、SSH環境の選択

    一方、次はCLIの方が優れている

    • スクリプトによる自動化
    • Agent SDK
    • すべてのCLIフラグ
    • 一部の高度なコマンド
    • Agent Teams
    • 詳細なサーバーモード設定

    Desktopには統合ターミナルもあるが、ターミナルを使わずGUIだけで大部分の作業ができる。Claude Desktop Code公式ドキュメント

    5

    iPhoneアプリ版

    iPhoneアプリは、Mac miniで動いているClaude Codeを操作する「リモコン」に近いもの。

    できること

    • プロンプトを送る
    • 作業の進捗を見る
    • ファイル編集やコマンド実行を承認する
    • 画像やファイルを添付する
    • 完了通知や承認依頼を受け取る
    • セッション名を変更する
    • /compact、/clear、/exit等を実行する
    • クラウドセッションを新しく開始する

    不得意・できないこと

    • コード全体をエディタとして閲覧・編集する
    • 対話式の/resume選択画面を使う
    • Remote Controlサーバーを起動する
    • CLIの詳細設定
    • MCPやプラグインの初期構築
    • 終了済みローカルセッションを確実に復旧する
    実行はMac mini上で行われる

    Remote Control利用中は、iPhoneで操作していても、コードの実行とファイル操作はMac mini上で行われる。Remote Control公式ドキュメント

    6

    今回のおすすめ構成

    1つに統一する必要はない。役割分担が最も使いやすい。

    Mac mini

    • CLI：Remote Controlサーバーの起動と復旧専用
    • Claude Desktop：必要に応じて差分確認やプレビュー
    • 実際のコードとGitを保存する本体

    MacBook Air

    • 普段はClaude DesktopまたはWeb版
    • Mac miniのプロジェクト別Remote Control環境を選択
    • コードを細かく確認したくなったらVS Codeを使用
    • MacBook Airのローカル環境を誤って選ばない

    iPhone

    • Remote Controlで進捗確認
    • 指示、承認、方向修正
    • 作業終了時のSTATUS.md更新指示
    • 長時間処理の完了通知を受信

    つまり、現状の最適解は次の構成。

    CLI ＝裏側のサーバー・保守 Claude Desktop／Web ＝普段のメイン操作画面 IDE ＝コードを細かく確認・手直しするとき iPhoneアプリ ＝外出先からの指示・確認・承認
    結論

    ターミナルを日常的に使いたくないという希望を考えると、普段はMacBook AirのClaude Desktop＋iPhoneアプリ、Mac miniのCLIはサーバー起動時だけが最も合っている。

    内蔵ターミナルでCLIは使えるか

    IDEやClaude Desktopの「内蔵ターミナル」で、スタンドアロン版Claude Code CLIがインストールされていれば、基本的に通常のターミナルと同じCLI機能を使える。ただし、2点重要な注意がある。

    内蔵ターミナルとチャット画面は別物

    操作場所CLIと同じ機能
    VS CodeのClaudeチャットパネル一部のみ
    VS Codeの内蔵ターミナルでclaudeを実行ほぼ全部
    Claude DesktopのCodeチャット画面一部はGUIに置き換え、一部非対応
    Claude Desktopの内蔵ターミナルでclaudeを実行ほぼ全部
    iPhone／WebのRemote Control画面モバイル対応コマンドのみ

    VS Code拡張には内部用のClaude Codeが含まれるが、内蔵ターミナルでclaudeを実行するには、スタンドアロンCLIのインストールが必要。VS Code版公式ドキュメント

    確認方法は次のとおり。

    BASH
    which claude
claude --version

    結果が表示されれば、その内蔵ターミナルからCLI版を使用できる。

    「どのMacで動いているターミナルか」に注意

    内蔵ターミナルで実行した処理は、原則としてそのアプリを開いているMacで動く。

    操作実行場所
    Mac miniのClaude Desktop内蔵ターミナルMac mini
    Mac miniのVS Code内蔵ターミナルMac mini
    MacBook AirのVS Code内蔵ターミナルMacBook Air
    MacBook AirのClaude Desktop内蔵ターミナルMacBook Air
    VS CodeでRemote SSH接続したMac miniのターミナルMac mini
    iPhoneからMac miniのRemote Controlを操作Mac mini

    MacBook AirのClaude DesktopでMac miniのRemote Controlセッションを見ていても、MacBook Airのローカルターミナルが自動的にMac miniのターミナルになるわけではない。

    常時稼働サーバーは通常のターミナルで

    24時間稼働させるRemote Controlサーバーは、IDEやDesktopを終了すると内蔵ターミナルも終了する可能性がある。したがって、Mac miniの常時稼働サーバーは、当面は通常のターミナルか、将来的にはLaunchAgentによる自動起動が適している。

    各機能で具体的にできること

    1. claude remote-control --capacity 3

    これは、Mac miniをClaude Codeの遠隔作業サーバーとして待機させる機能。

    BASH
    cd ~/Developer/hotel-app
claude remote-control \
  --name "hotel-app-main" \
  --spawn same-dir \
  --capacity 3 \
  --sandbox

    実現できること

    • iPhoneからMac mini上のコードを変更する
    • MacBook AirのWeb／アプリから指示する
    • Mac mini上でテスト、ビルド、Git操作を実行する
    • 最大3つのClaude Codeセッションを同時に保持する
    • Mac miniに設定したMCPや開発ツールを遠隔利用する
    • 各端末で同じ会話・進捗を見る
    3の意味

    ここでいう3は、端末数やタスク数ではなく、Mac miniで同時に保持できるClaude Codeセッション数。今回の運用では、最も重要なCLI機能。Remote Control公式ドキュメント

    2. claude -cとclaude --resume

    終了・切断したセッションを復旧する機能。

    直近のセッションを再開

    BASH
    cd ~/Developer/hotel-app
claude -c

    例えば、次の場合に使う。

    • Mac miniを再起動した
    • Claude Codeを終了した
    • ターミナルを閉じてしまった
    • 前日の作業を再開したい

    履歴一覧から選ぶ

    BASH
    cd ~/Developer/hotel-app
claude --resume

    表示された履歴から、再開したいセッションを選ぶ。

    名前で直接再開

    BASH
    claude --resume hotel-app__main

    実現できること

    • 過去の会話を維持したまま作業再開
    • Claudeが以前読んだファイルや決定事項を引き継ぐ
    • 以前のRemote Controlへ再接続
    • プロジェクトごとに履歴を絞って探す

    これはMac miniの復旧操作として重要。

    3. MCPサーバーの追加・設定

    MCPは、Claude Codeに外部サービスを操作するための「専用アダプター」を追加する仕組み。通常のClaude Codeは、主にローカルファイルやターミナルを扱う。MCPを追加すると、外部サービスまで操作範囲を広げられる。

    GitHub MCP

    • Issueを読む
    • Pull Requestを作る
    • PRのレビューコメントを確認する
    • CIの失敗結果を取得する

    データベースMCP

    • SupabaseやPostgreSQLのテーブル構成を確認する
    • SQLを実行する
    • データの不整合を調査する
    本番データベース

    本番データベースには、原則として読み取り専用権限から始める。

    Sentry等のエラー監視MCP

    • 本番環境のエラーを取得する
    • エラーが発生したコードを特定する
    • 修正案を作る

    Figma等のデザインMCP

    • デザイン情報を読み取る
    • 色、余白、画面構成をコードへ反映する

    社内サービス

    • Notionの仕様書を読む
    • Google Driveの資料を確認する
    • Slackの障害報告を取得する

    MCPの初期追加はCLIが最も確実。追加後は、CLI、IDE、Desktopのローカルセッションで設定を共有できる。初心者の場合、最初に導入するならGitHub MCPが候補。データベースや外部送信を伴うMCPは、権限範囲を理解してから導入する。

    4. スラッシュコマンドと設定

    スラッシュコマンドは、Claudeへの質問ではなく、Claude Code自体を操作する命令。よく使うものは次のとおり。

    コマンドできること
    /renameセッションへ名前を付ける
    /resume過去のセッションへ切り替える
    /compact長い会話を要約して継続する
    /context会話容量の使用状況を確認する
    /model使用モデルを変更する
    /permissionsClaudeに許可する操作を設定する
    /diffClaudeが変更したコードを確認する
    /rewind会話やコードを以前の状態へ戻す
    /branch現在の会話から別案を分岐する
    /mcpMCPの接続状態を確認する
    /pluginプラグインを管理する
    /doctorClaude Codeの設定・不具合を診断する
    /remote-control現在のセッションを遠隔操作可能にする
    /export会話内容をファイルへ出力する
    /exitセッションを終了する

    CLIでは原則としてすべて利用できる。VS Codeのチャットパネルは一部のみ、Desktopでは一部がボタンや設定画面に置き換えられている。例えばDesktopで/configを実行すると、テキスト設定ではなく設定画面が開く。

    初心者が最初に覚えるべきものは、次の6つ。

    最初に覚える6つ
    /rename
/compact
/context
/diff
/doctor
/exit

    5. シェルスクリプト

    シェルスクリプトは、複数のターミナル操作を「決められた手順」として保存するもの。例えば、通常は毎回次の操作が必要だとする。

    1. プロジェクトフォルダへ移動 2. 最新コードを取得 3. パッケージをインストール 4. テストを実行 5. 結果を保存

    これを1つのスクリプトにすると、1回の実行で順番に処理できる。

    実現例

    • すべてのテストを実行する
    • アプリをビルドする
    • データベースをバックアップする
    • Gitの状態を記録する
    • ログファイルを整理する
    • Remote Controlを決まった設定で起動する
    • Claude Codeへ分析を依頼する

    シェルスクリプトは、決められた操作を毎回同じように実行するのが得意。一方、状況を判断して対応を変える部分にはClaudeを組み合わせる。

    6. 定期実行

    Claude Codeには複数の定期実行方法がある。

    /loop

    現在のセッション内で、一定間隔ごとに確認させる。

    コマンド
    /loop 5m デプロイが完了したか確認してください
    • デプロイ完了を5分ごとに確認
    • CIの結果を定期確認
    • 長時間ビルドの終了を監視
    • PRに新しいコメントが付いたか確認

    セッションが動いている間の短期監視向け。

    Claude DesktopのScheduled Tasks

    GUIから設定できる。例えば：

    • 毎朝9時にプロジェクトのエラーを確認
    • 毎週月曜に依存パッケージを確認
    • 毎日、未コミット変更を報告
    • 毎晩テストを実行して結果をまとめる

    ローカルタスクはMacが起動し、Claude Desktopが開いている必要がある。ローカルファイルや未コミット変更を扱える。Desktop定期実行公式ドキュメント

    クラウドRoutines

    Anthropicのクラウド上で実行する。

    • Mac miniがオフでも動く
    • 毎朝GitHubリポジトリを確認する
    • 定期的にPRレビューを実行する
    • GitHubイベントをきっかけに処理する

    ただし、Mac miniだけにある未コミットファイルにはアクセスできない。

    7. Agent SDK

    Agent SDKは、Claude Codeのように自律的に作業するAIを、自分のアプリへ組み込むための開発キット。PythonまたはTypeScriptから利用する。通常のClaude APIは、質問を送って回答を受け取るのが中心。Agent SDKでは、Claudeに以下を行わせられる。

    • ファイルを読む
    • コードを書く
    • コマンドを実行する
    • Web検索する
    • MCPを使う
    • 複数ステップの作業を自律的に進める
    • 承認を求める
    • 結果を決められたJSON形式で返す

    バグ修正エージェント

    社内画面にエラー内容を入力すると、次まで行う。

    1. 関係するコードを探す
    2. 原因を調べる
    3. 修正する
    4. テストする
    5. Pull Requestを作る

    自動コードレビュー

    GitHubへコードがpushされたら、次を行う。

    1. 変更内容を読む
    2. セキュリティ上の問題を確認
    3. バグの可能性を指摘
    4. レビュー結果をPRへ投稿する

    社内業務エージェント

    例えば宿泊施設向けアプリに、次の機能を組み込める。

    • 問い合わせ内容を分類する
    • 過去の対応履歴を調べる
    • 返信案を作る
    • 担当部署を判定する
    • 人間の承認後に送信する
    上級者向け

    Agent SDKは上級者向け。PythonまたはTypeScriptの実装が必要、API利用料が発生、実行環境の用意が必要、ファイル・外部送信・データベース権限の設計が必要、暴走防止と承認フローが必要。現時点では使わなくても問題ない。Agent SDK公式概要

    8. スクリプトによるClaudeの自動化

    これは、対話画面を開かずにClaudeへ1回の仕事を依頼する機能。

    BASH
    claude -p "テストを実行し、失敗原因をまとめてください。ファイルは変更しないでください"

    -pは「質問を実行し、結果を出力して終了する」という意味。

    実現例

    • テスト失敗を自動分析
    • Git差分を毎日要約
    • リリースノートを自動作成
    • 複数のREADMEをまとめて更新
    • CIでPull Requestを自動レビュー
    • エラーログを読み、原因候補をJSONで返す
    • 定型的なコード修正を自動実行

    使用できる道具を限定することもできる。

    BASH
    claude -p \
  "テストを実行し、失敗原因をまとめてください。変更はしないでください" \
  --allowedTools "Read,Bash"

    この場合、ファイルの読み取りとコマンド実行だけを許可し、編集は許可しない。シェルスクリプトとの違いは次のとおり。

    • シェルスクリプト：決められた処理をそのまま実行
    • Claude自動化：状況を読んで判断し、結果を変える

    プログラムからClaude Codeを実行する公式説明

    9. CLIフラグ

    CLIフラグは、Claude Codeを起動するときに付ける追加条件。代表例：

    フラグ実現できること
    --nameセッション名を指定
    --continue直近セッションを継続
    --resumeセッションを指定して再開
    --remote-control遠隔操作を有効化
    --model使用モデルを指定
    --permission-mode plan計画だけ作り、承認まで編集させない
    --allowedTools使用を許可する道具を限定
    --disallowedTools使用禁止の道具を指定
    --worktree別の作業用Gitフォルダで安全に並行作業
    --fork-session会話を複製して別案へ分岐
    --max-turns自律的に進める回数を制限
    --output-format json結果をプログラムが読める形式で出力
    --mcp-config使用するMCP設定を指定
    --add-dir別フォルダへのアクセスを追加

    例えば、安全に調査だけさせる場合：

    BASH
    claude \
  --permission-mode plan \
  --name "hotel-app-investigation"

    GUIにもモデル選択やPermission Modeはあるが、すべての細かい起動条件を組み合わせられるのはCLI。

    10. 一部の高度なコマンド

    代表的なものは次のとおり。

    /rewind

    Claudeが変更する前の状態へ戻す。

    誤った修正を行った → /rewind → 戻したい時点を選ぶ

    /branch

    現在の会話履歴を残したまま、別の方針を試す。

    現在の方式は残したい 別の技術でも試したい → /branch

    claude agents

    複数のバックグラウンドセッションを一覧管理する。作業中／入力待ち／完了／停止済みを1画面で確認できる。

    /teleport

    クラウドで作ったセッションをローカルCLIへ取り込む。

    /export

    会話履歴をテキストファイルとして保存する。

    /doctor

    インストール、ログイン、設定、接続の問題を診断する。

    初心者がまず覚えるなら

    初心者の場合、まず覚えるなら/doctor、/diff、/rewindの3つが有用。

    11. Agent Teams

    Agent Teamsは、複数のClaude Codeをチームとして働かせる機能。例えば1つのWebアプリ開発を、次のように分担できる。

    リーダーClaude ├── フロントエンド担当Claude ├── バックエンド担当Claude ├── テスト担当Claude └── セキュリティレビュー担当Claude

    リーダーが仕事を分割し、各Claudeへ割り振る。担当Claude同士もメッセージを送り、共有タスクリストを使って調整する。

    実現できること

    • フロントエンド・バックエンド・テストを並行開発
    • 複数の原因仮説を同時に調査
    • 実装担当とレビュー担当を分離
    • 大規模なコード調査を分担
    • 各担当の結果をリーダーが統合

    ただし、現在は実験的機能で、初期状態では無効。

    JSON
    {
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
    注意点

    通常の1セッションより大幅にトークンを使う／同じファイルを編集すると競合する／タスクの分割方法が悪いと逆に遅くなる／セッション復旧や終了処理に既知の制約がある／DesktopではなくCLI中心の機能／Git、テスト、worktreeの理解が必要。

    Agent Teamsが向いているのは、フロントエンド、バックエンド、テストのように、作業範囲を明確に分けられる場合。同じファイルを順番に直す仕事には向かない。Agent Teams公式ドキュメント

    現段階では、Agent Teamsよりも、まず1つのメインセッションと必要に応じたSubagentの利用をおすすめする。
